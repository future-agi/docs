## Gateway

The gateway is the core of Prism — a high-performance proxy that handles every LLM request. It processes requests through a plugin pipeline:

```
Request ──> Auth ──> Rate Limit ──> Guardrails (Pre) ──> Cache ──> Provider ──> Guardrails (Post) ──> Response
```

Each plugin runs at a fixed priority, adding minimal overhead per request.

## Virtual API Keys

Prism uses **virtual API keys** (prefixed `sk-prism-`) to authenticate requests. These are not your provider API keys — they're Prism-specific keys that map to an organization and its configuration.

```bash
# This is a Prism virtual key, not an OpenAI key
curl https://prism.futureagi.com/v1/chat/completions \
  -H "Authorization: Bearer sk-prism-abc123..."
```

When a request arrives, Prism:

1. Validates the virtual key
2. Resolves which organization the key belongs to
3. Loads that org's provider credentials, guardrails, routing rules, and rate limits
4. Routes the request to the appropriate LLM provider using the org's own API keys

<Tip>
Virtual keys keep your provider API keys secure. Your application never sees or stores raw provider credentials — Prism manages them server-side.
</Tip>

## Organizations & Multi-Tenancy

Prism supports **multi-tenant isolation** out of the box. Each organization gets its own:

| Resource | Isolation |
|----------|-----------|
| **Providers** | Each org configures its own provider API keys |
| **Guardrails** | Org-specific safety rules and policies |
| **Routing** | Independent load balancing and failover policies |
| **Rate Limits** | Per-org and per-key rate limiting |
| **Budgets** | Spend limits and alerts scoped to the org |
| **Caching** | Isolated cache entries |

The organization is resolved from the virtual API key on every request. Prism serves multiple organizations simultaneously with full isolation.

### Configuration Hierarchy

Settings are resolved in this order (most specific wins):

```
Request Headers  >  API Key Config  >  Org Config  >  Global Config
```

For example, a request can override the cache TTL with an `x-prism-cache-ttl` header, even if the org config sets a different default.

## Providers

A **provider** is an LLM service that Prism routes requests to. Each provider has:

- **Name** — e.g., `openai`, `anthropic`, `gemini`
- **API format** — How to translate requests (`openai`, `anthropic`, `gemini`, `bedrock`, `azure`, `cohere`)
- **Base URL** — The provider's API endpoint
- **API key** — The provider's credentials (stored encrypted)
- **Models** — Which models are available through this provider

Providers with an OpenAI-compatible API (Groq, Together AI, Fireworks, Mistral, etc.) use `api_format: "openai"` and work with zero translation.

## Org Configuration

Each organization has a versioned **org config** that controls all gateway behavior. The config includes:

| Section | What it controls |
|---------|-----------------|
| `providers` | Which LLM providers and models are available |
| `guardrails` | Safety rules (PII, injection, moderation, custom) |
| `routing` | Load balancing strategy, failover, retries |
| `cache` | Caching mode, TTL, max entries |
| `rate_limiting` | Requests per minute/hour/day limits |
| `budgets` | Spend limits and alert thresholds |
| `cost_tracking` | Cost calculation and attribution settings |
| `ip_acl` | IP allowlist/blocklist |
| `alerting` | Email/webhook alerts for events |

Configs are **versioned** — every change creates a new version. You can roll back to any previous version.

<Note>
Changes to org config are pushed to the gateway in real-time. No gateway restart needed.
</Note>

## Guardrails

Guardrails are safety checks that run on requests (pre-processing) and responses (post-processing). Prism includes 18+ built-in guardrail types:

| Type | Stage | What it catches |
|------|-------|-----------------|
| PII Detection | Pre & Post | Email addresses, SSNs, credit cards, phone numbers, API keys |
| Prompt Injection | Pre | Attempts to override system instructions |
| Content Moderation | Pre & Post | Harmful, offensive, or inappropriate content |
| Secret Detection | Pre & Post | API keys, passwords, tokens in messages |
| Hallucination Detection | Post | Factually inconsistent or unsupported claims |
| System Prompt Protection | Pre | Attempts to extract or reveal the system prompt |
| Topic Restriction | Pre | Off-topic requests |
| Language Detection | Pre | Non-allowed languages or scripts |
| Input Validation | Pre | Messages exceeding length limits or matching denied patterns |
| Tool Permissions | Pre & Post | Unauthorized tool or function calls |
| Custom Expression Rules | Pre & Post | Your own rules using Prism's expression language |
| Webhook | Pre & Post | Custom guardrail logic via external HTTP endpoint |

Guardrails can run in three modes:

- **Enforce** — Block the request (returns HTTP 446)
- **Monitor** — Log a warning but let the request through (returns HTTP 246 with a warning header)
- **Log** — Silently record the guardrail result with no user-facing impact (useful for shadow-testing new rules)

## Sessions

Sessions group related requests together for tracking and analytics. To use sessions:

1. Create a session via the dashboard or API
2. Pass the session ID in your requests via the `x-prism-session-id` header

```bash
curl $PRISM_BASE_URL/v1/chat/completions \
  -H "Authorization: Bearer $PRISM_API_KEY" \
  -H "x-prism-session-id: my-session-123" \
  -H "Content-Type: application/json" \
  -d '{"model": "gpt-4o-mini", "messages": [...]}'
```

<Note>
Sessions are for grouping and analytics only. Prism does not maintain conversation state — your application must send the full message history with each request.
</Note>

## Custom Metadata

Attach arbitrary metadata to any request using the `x-prism-metadata` header. This metadata appears in logs and can be used for filtering and attribution:

```bash
curl $PRISM_BASE_URL/v1/chat/completions \
  -H "Authorization: Bearer $PRISM_API_KEY" \
  -H 'x-prism-metadata: {"user_tier": "premium", "feature": "search"}' \
  -H "Content-Type: application/json" \
  -d '{"model": "gpt-4o-mini", "messages": [...]}'
```

## Request Headers

These headers let you control Prism behavior on a per-request basis:

| Header | Description |
|--------|-------------|
| `x-prism-session-id` | Group requests into a session |
| `x-prism-metadata` | Attach custom metadata (JSON) |
| `x-prism-trace-id` | Set a custom trace ID for distributed tracing |
| `x-prism-cache-ttl` | Override cache TTL for this request (seconds) |
| `x-prism-cache-force-refresh` | Force cache miss (`true`/`false`) |
| `Cache-Control: no-store` | Disable caching for this request |
