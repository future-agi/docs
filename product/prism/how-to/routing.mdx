## Overview

Prism's routing layer distributes requests across multiple providers and models to maximize reliability and performance. If one provider is down or slow, traffic automatically shifts to healthy alternatives.

---

## Routing Strategies

| Strategy | How it works |
|----------|-------------|
| **Round Robin** | Distributes requests evenly across providers in rotation |
| **Weighted** | Routes based on assigned weights (e.g., 70% OpenAI, 30% Anthropic) |
| **Latency-based** | Routes to the fastest provider based on recent response times |
| **Cost-based** | Routes to the cheapest provider that supports the requested model |
| **Priority** | Uses providers in order — falls back to the next only on failure |

## Failover

When a provider returns an error, Prism can automatically retry with an alternative provider. Failover triggers on:

- **HTTP 429** — Rate limited
- **HTTP 5xx** — Server errors
- **Timeout** — Provider didn't respond in time
- **Connection errors** — Provider unreachable

```
Request ──> OpenAI (429) ──> Anthropic (200) ──> Response
                 │                  ▲
                 └── failover ──────┘
```

Failover is configured as part of the routing policy:

```json
{
  "strategy": "priority",
  "config": {
    "providers": ["openai", "anthropic", "gemini"],
    "failover_on": [429, 500, 502, 503, 504]
  }
}
```

## Retries

Prism retries failed requests with exponential backoff before failing over to another provider:

| Setting | Default | Description |
|---------|---------|-------------|
| `max_retries` | 2 | Maximum retry attempts per provider |
| `initial_backoff_ms` | 100 | Initial wait before first retry |
| `max_backoff_ms` | 5000 | Maximum backoff duration |
| `backoff_multiplier` | 2 | Multiplier applied after each retry |

```json
{
  "routing": {
    "retries": {
      "max_retries": 3,
      "initial_backoff_ms": 200,
      "max_backoff_ms": 10000,
      "backoff_multiplier": 2
    }
  }
}
```

## Circuit Breaking

If a provider consistently fails, Prism's circuit breaker temporarily stops sending requests to it — preventing cascading failures and wasted latency.

| State | Behavior |
|-------|----------|
| **Closed** | Normal operation — requests flow to the provider |
| **Open** | Provider is unhealthy — all requests bypass it |
| **Half-open** | Testing — a small percentage of requests are sent to check recovery |

The circuit breaker transitions based on error rate thresholds:

```json
{
  "routing": {
    "circuit_breaker": {
      "error_threshold_percent": 50,
      "min_requests": 10,
      "open_duration_seconds": 30,
      "half_open_max_requests": 3
    }
  }
}
```

<Tip>
Circuit breaking works together with failover. When a provider's circuit is open, requests automatically route to the next available provider.
</Tip>

## Timeouts

Set per-request and per-provider timeouts to prevent slow requests from blocking your application:

```json
{
  "routing": {
    "timeout_ms": 30000,
    "per_provider_timeout_ms": {
      "openai": 25000,
      "anthropic": 30000
    }
  }
}
```

The `X-Prism-Timeout-Ms` response header shows the timeout that was applied.

## Example: High-Availability Setup

A production routing config combining all reliability features:

```json
{
  "routing": {
    "strategy": "priority",
    "config": {
      "providers": ["openai", "anthropic", "gemini"],
      "failover_on": [429, 500, 502, 503, 504]
    },
    "retries": {
      "max_retries": 2,
      "initial_backoff_ms": 100,
      "max_backoff_ms": 5000
    },
    "circuit_breaker": {
      "error_threshold_percent": 50,
      "min_requests": 10,
      "open_duration_seconds": 30
    },
    "timeout_ms": 30000
  }
}
```

This config:
1. Routes to OpenAI first
2. If OpenAI fails (429/5xx), retries twice with backoff
3. If retries are exhausted, fails over to Anthropic
4. If Anthropic also fails, fails over to Gemini
5. If OpenAI's error rate exceeds 50%, the circuit breaker opens and skips it entirely for 30 seconds
