## What Are Guardrails?

Guardrails are safety checks that run on every request and response flowing through Prism. They catch dangerous or unwanted content **before** it reaches the LLM (pre-processing) or **before** it reaches your users (post-processing).

---

## Built-in Guardrail Types

Prism ships with 18+ guardrail engines:

| Guardrail | Stage | What it detects |
|-----------|-------|-----------------|
| **PII Detection** | Pre & Post | Email addresses, SSNs, credit card numbers, phone numbers, API keys |
| **Prompt Injection** | Pre | Attempts to override system instructions, jailbreak prompts |
| **Content Moderation** | Pre & Post | Hate speech, threats, harassment, sexual content, violence |
| **Secret Detection** | Pre & Post | API keys, tokens, passwords, private keys in messages |
| **Hallucination Detection** | Post | Factually inconsistent or unsupported claims |
| **Topic Restriction** | Pre | Off-topic requests outside allowed subjects |
| **Language Detection** | Pre | Messages in non-allowed languages |
| **Data Leakage Prevention** | Post | Sensitive data in model responses |
| **Blocklist** | Pre & Post | Custom word/phrase blocklists |
| **System Prompt Protection** | Pre | Attempts to extract or reveal the system prompt |
| **Tool Permissions** | Pre & Post | Unauthorized tool or function calls (allowlist/denylist) |
| **Input Validation** | Pre | Messages exceeding length limits or matching denied patterns |
| **MCP Security** | Pre & Post | Unauthorized MCP tool calls, injection in tool inputs/outputs |
| **Custom Expression Rules** | Pre & Post | Your own rules using Prism's expression language |
| **Webhook (BYOG)** | Pre & Post | Call your own guardrail service via webhook |
| **Future AGI Evaluation** | Pre & Post | ML-based safety checks via the Future AGI evaluation API |

## Enforcement Modes

Each guardrail can operate in one of three modes:

| Mode | HTTP Status | Behavior |
|------|-------------|----------|
| **Enforce** | `403` | Block the request entirely. Returns an error response with type `guardrail_error`. |
| **Monitor** | `200` | Log a warning but let the request through. Guardrail results are recorded in logs. |
| **Log** | `200` | Silently log the guardrail result. No user-facing impact — useful for shadow-testing new rules. |

<Tip>
Start with **Monitor** mode to understand your traffic patterns before switching to **Enforce**. This prevents false positives from blocking legitimate requests.
</Tip>

## Setting Up Guardrails

### Via Dashboard

1. Navigate to **Prism > Guardrails** in the [Future AGI dashboard](https://app.futureagi.com)
2. Click **Add Guardrail Policy**
3. Select the guardrail type (e.g., PII Detection)
4. Choose the mode: **Enforce** or **Monitor**
5. Configure type-specific settings (e.g., which PII patterns to detect)
6. Set the scope: **Global**, **Project**, or **API Key**
7. Click **Save**

### Via Python SDK

Install the SDK:

```bash
pip install prism-ai
```

Create a guardrail config and attach a policy:

```python
from prism import Prism

client = Prism(
    api_key="sk-prism-your-key",
    base_url="https://prism.futureagi.com",
    control_plane_url="https://api.futureagi.com",
)

# Create a guardrail config with multiple rules
config = client.guardrails.configs.create(
    name="Production Safety",
    rules=[
        {
            "name": "pii-detector",
            "stage": "pre",
            "action": "block",
            "threshold": 0.5,
            "config": {"entities": ["EMAIL", "SSN", "CREDIT_CARD", "PHONE"]},
        },
        {
            "name": "injection-detector",
            "stage": "pre",
            "action": "warn",
            "threshold": 0.6,
        },
        {
            "name": "content-moderation",
            "stage": "pre",
            "action": "block",
            "threshold": 0.7,
        },
        {
            "name": "secrets-detector",
            "stage": "pre",
            "action": "block",
            "threshold": 0.5,
        },
    ],
    fail_open=False,
)

# Create a policy to apply the config
policy = client.guardrails.policies.create(
    name="Apply to all keys",
    guardrail_config_id=config["id"],
    scope="gateway",
)
```

## Testing Guardrails

Once your guardrail policy is active, guardrails are applied automatically to all requests routed through Prism.

<Note>
The Prism gateway URL is `https://prism.futureagi.com`. In the curl examples below, `$PRISM_BASE_URL` refers to this URL and `$PRISM_API_KEY` is your Prism API key (e.g., `sk-prism-...`).
</Note>

### PII Detection

<CodeGroup>

```python Python SDK
from prism import Prism

client = Prism(
    api_key="sk-prism-your-key",
    base_url="https://prism.futureagi.com",
)

# Should be BLOCKED — contains PII
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "My SSN is 123-45-6789 and email is test@example.com"}],
)
```

```bash curl
# Should be BLOCKED (403) — contains PII
curl -s -o /dev/null -w "%{http_code}" $PRISM_BASE_URL/v1/chat/completions \
  -H "Authorization: Bearer $PRISM_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o-mini",
    "messages": [{"role": "user", "content": "My SSN is 123-45-6789 and email is test@example.com"}]
  }'
# Expected: 403
```

</CodeGroup>

## PII Remediation Modes

The PII guardrail supports multiple remediation strategies beyond simply blocking:

| Mode | Behavior | Example |
|------|----------|---------|
| **Block** | Reject the request entirely | Request returns `403` |
| **Mask** | Replace PII with asterisks | `test@example.com` → `***` |
| **Redact** | Replace with a typed placeholder | `test@example.com` → `[REDACTED:email]` |
| **Hash** | Replace with a truncated SHA hash | `test@example.com` → `[SHA:a1b2c3d4]` |

Configure the remediation mode in your guardrail config:

```python
from prism import Prism

client = Prism(
    api_key="sk-prism-your-key",
    base_url="https://prism.futureagi.com",
    control_plane_url="https://api.futureagi.com",
)

config = client.guardrails.configs.create(
    name="Redact PII in production",
    rules=[
        {
            "name": "pii-detector",
            "stage": "pre",
            "action": "redact",
            "threshold": 0.5,
            "config": {
                "entities": ["EMAIL", "SSN", "CREDIT_CARD", "PHONE"],
                "remediation": "redact",
            },
        },
    ],
    fail_open=False,
)

policy = client.guardrails.policies.create(
    name="Redact PII globally",
    guardrail_config_id=config["id"],
    scope="gateway",
)
```

<Tip>
Use **Redact** or **Mask** mode to sanitize PII while still allowing the request through. This is useful when you want to protect sensitive data without blocking legitimate requests.
</Tip>

## Score Thresholds

Every guardrail returns a confidence score between `0.0` (safe) and `1.0` (maximum violation). You can configure a threshold per guardrail to control sensitivity:

```json
{
  "name": "pii-detector",
  "action": "block",
  "threshold": 0.7
}
```

A guardrail triggers when `score > threshold`. Lower thresholds catch more violations but may increase false positives. Higher thresholds reduce noise but may miss edge cases.

| Threshold | Sensitivity | Use case |
|-----------|-------------|----------|
| `0.3` | High | Strict compliance environments |
| `0.5` | Medium | General production use (default) |
| `0.8` | Low | Permissive — only flag high-confidence violations |

## Streaming Guardrails

Guardrails work with streaming responses too. Post-processing guardrails accumulate streamed chunks and run safety checks on the assembled output. If a guardrail triggers mid-stream, the stream is terminated with an error event.

## Custom Blocklists

### Via Dashboard

1. Navigate to **Prism > Guardrails > Blocklists** in the [Future AGI dashboard](https://app.futureagi.com)
2. Click **Create Blocklist**
3. Enter a name and add your blocked words or phrases
4. Click **Save**
5. Reference the blocklist in a guardrail policy to block or flag requests containing those terms

### Via Python SDK

```python
from prism import Prism

client = Prism(
    api_key="sk-prism-your-key",
    base_url="https://prism.futureagi.com",
    control_plane_url="https://api.futureagi.com",
)

# Create a blocklist
blocklist = client.guardrails.blocklists.create(
    name="Competitor names",
    words=["CompetitorA", "CompetitorB"],
)

# Add words to an existing blocklist
client.guardrails.blocklists.add_words(
    blocklist_id=blocklist["id"],
    words=["CompetitorC"],
)
```

Reference the blocklist in a guardrail policy to block or flag requests containing those terms.

## Guardrail Feedback

When a guardrail blocks or flags a request, you can provide feedback on whether the decision was correct. This helps tune guardrail sensitivity over time.

### Via Python SDK

```python
from prism import Prism

client = Prism(
    api_key="sk-prism-your-key",
    base_url="https://prism.futureagi.com",
    control_plane_url="https://api.futureagi.com",
)

# Submit feedback on a guardrail decision
client.feedback.create(
    trace_id="<trace-id>",
    value=0.0,
    metadata={
        "type": "false_positive",
        "comment": "Legitimate request about data formats",
    },
)
```

The `trace_id` is returned in the response headers of any guardrail-flagged request. Feedback values range from `0.0` (incorrect guardrail decision) to `1.0` (correct decision). Use the `metadata` field to provide additional context such as the feedback type (`false_positive`, `false_negative`, `correct`) and an optional comment.
