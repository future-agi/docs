## What is Prism?

Prism is Future AGI's **AI Gateway** — a high-performance proxy layer that sits between your application and LLM providers. It exposes a single **OpenAI-compatible API** while adding routing, safety guardrails, cost management, caching, rate limiting, and observability.

```
Your App  ──>  Prism Gateway  ──>  OpenAI / Anthropic / Gemini / Bedrock / ...
               │
               ├── Auth (virtual API keys)
               ├── Guardrails (PII, injection, moderation)
               ├── Routing (load balance, failover)
               ├── Caching (exact + semantic)
               ├── Cost tracking
               ├── Logging & tracing
               └── Rate limiting
```

Change your `base_url`, keep everything else. Every provider's response is normalized to the OpenAI format — write code once, switch providers freely.

## Architecture

Prism has three layers:

| Layer | Role |
|-------|------|
| **Gateway** | The data plane. Handles every LLM request with minimal overhead. Routing, caching, streaming, guardrails, and cost tracking all happen here. |
| **Control Plane** | Management APIs. API key lifecycle, org configuration, analytics aggregation, webhook delivery. Runs separately from the gateway — not involved in processing individual LLM requests. |
| **Dashboard** | Web UI for request logs, analytics, provider management, guardrail configuration, and budget tracking. |

### Request Pipeline

Every request flows through a plugin pipeline in the gateway:

```
Request ──> IP ACL ──> Auth ──> RBAC ──> Cache Lookup
                                             │
                               [hit] ◄──────┤──────► [miss]
                                 │                      │
                             Return cached     Budget ──> Guardrails (Pre)
                                               ──> Validation ──> Rate Limit
                                               ──> Provider Call
                                                        │
                                           Guardrails (Post) ──> Cost ──> Logging ──> Response
```

## Response Headers

Every response from Prism includes metadata headers. Some headers are always present; others only appear when a specific feature is active.

### Always Present

| Header | Description |
|--------|-------------|
| `X-Prism-Request-Id` | Unique request ID (ULID) for log correlation |
| `X-Prism-Trace-Id` | Trace ID for distributed tracing |
| `X-Prism-Latency-Ms` | Total latency including provider call |
| `X-Prism-Model-Used` | Actual model used (may differ due to routing) |
| `X-Prism-Provider` | Provider that served the request |
| `X-Prism-Timeout-Ms` | Timeout applied to the request |

### Conditional

| Header | Present when |
|--------|-------------|
| `X-Prism-Cost` | Model has pricing data and cost can be calculated (not on cache hits) |
| `X-Prism-Cache` | Caching is enabled — value is `miss`, `hit`, or `skip` |
| `X-Prism-Guardrail-Triggered` | A guardrail policy is active and was triggered — value is `true` |
| `X-Prism-Fallback-Used` | A provider fallback occurred — value is `true` |
| `X-Prism-Routing-Strategy` | A routing policy is active — value is e.g. `round-robin`, `weighted` |
| `X-Ratelimit-Limit-Requests` | Rate limiting is enabled — ceiling in requests per minute |
| `X-Ratelimit-Remaining-Requests` | Rate limiting is enabled — requests remaining in the current window |
| `X-Ratelimit-Reset-Requests` | Rate limiting is enabled — Unix timestamp when the limit resets |
