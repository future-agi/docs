## What is this page?

A **session** is a group of traces that belong to the same conversation or workflow. Instead of looking at every individual request your AI handled, sessions let you see whole multi-turn conversations as a single unit — first message, last message, total duration, total cost, and how many traces were part of it.

Sessions are created automatically when traces carry a `session.id` attribute. All traces with the same session ID in the same project get grouped together.

<img src="/images/docs/observe/sessions-overview.png" alt="Sessions tab overview" style={{ borderRadius: '5px' }} />

---

## When would you use this?

- **Reviewing a chatbot conversation** — See the whole exchange end-to-end instead of one request at a time.
- **Analyzing a user journey** — Group one user's sequence of requests as a session to find drop-off points.
- **Cost or duration per conversation** — See total cost and duration for an entire session, not just per trace.
- **Replaying production traffic** — Re-run real sessions through your latest agent to test changes.
- **Debugging a reported issue** — Open the session for that user's complaint and inspect every turn.

## The sessions list

The **Sessions** tab shows one row per session. Here are the columns (confirmed from the live platform):

| Column | What it tells you |
|---|---|
| **Session Id** | The unique identifier for this conversation (set via `session.id` on your spans). |
| **First Message** | The opening message of the conversation. |
| **Last Message** | The most recent message. |
| **Duration** | How long the conversation lasted. |
| **Total Cost** | Combined cost across every trace in this session. |
| **Total Traces** | How many individual requests were part of this session. |

A **Session Metrics** chart at the top plots latency and traffic over the selected time window.

## Session replay

Click **Replay Sessions** in the bulk actions dropdown to open the replay configuration dialog. The dialog walks you through three steps:

<img src="/images/docs/observe/sessions-replay-config.png" alt="Replay Sessions dialog with Create Scenarios, Start Replay, Map Eval Variables steps" style={{ borderRadius: '5px' }} />

<Steps>
  <Step title="Create Scenarios">
    Define the agent that will be replayed against. Fill in:
    - **Agent Definition** — the prompt/version that defines how your agent behaves.
    - **Scenario Details** — name for the scenario set, the number of sessions to replay, and the number of scenarios to generate.
  </Step>

  <Step title="Start Replay">
    Kick off the replay. The selected sessions are run through your agent definition.
  </Step>

  <Step title="Map Eval Variables">
    Connect any evaluation tasks you want run on the replayed sessions to specific input/output variables.
  </Step>
</Steps>

Replayed sessions appear back in the Tracing view as new traces, linked to the original session.

## How to associate traces with a session

For a trace to show up in a session, the span must carry a `session.id` attribute. There are two ways to set it:

<Steps>
  <Step title="Set session.id on a span directly">
    When creating a span manually, add the attribute:

    <CodeGroup>
    ```python Python
    from fi_instrumentation import register, FITracer
    from fi_instrumentation.fi_types import ProjectType
    from opentelemetry.trace import Status, StatusCode

    trace_provider = register(
        project_type=ProjectType.OBSERVE,
        project_name="PROJECT_NAME",
    )

    tracer = FITracer(trace_provider.get_tracer(__name__))

    with tracer.start_as_current_span("SPAN_NAME") as span:
        span.set_status(Status(StatusCode.OK))
        span.set_attribute("session.id", "session123")
        span.set_attribute("input.value", "user input here")
        span.set_attribute("output.value", "model response here")
    ```
    ```typescript JS/TS
    const { register, ProjectType } = require("@traceai/fi-core");

    const traceProvider = register({
        projectType: ProjectType.OBSERVE,
        projectName: "FUTURE_AGI"
    });

    const tracer = traceProvider.getTracer("manual-instrumentation-example");

    tracer.startActiveSpan("HandleFunctionCall", {}, (span) => {
        span.setAttribute("session.id", "my-session-id");
        span.end();
    });
    ```
    </CodeGroup>
  </Step>

  <Step title="Set session for many spans using context">
    To tag every span in a block with the same session ID:

    <CodeGroup>
    ```python Python
    from fi_instrumentation import using_session

    with using_session(session_id="my-session-id"):
        # All spans created within this block get session.id = "my-session-id"
        response = client.chat.completions.create(...)
    ```
    ```typescript JS/TS
    import { context, propagation } from "@opentelemetry/api";

    const sessionId = "my-js-session-id";

    const activeContext = context.active();
    const baggageWithSession = propagation.createBaggage({
        "session.id": { value: sessionId }
    });
    const newContext = propagation.setBaggage(activeContext, baggageWithSession);

    context.with(newContext, () => {
        // All spans created within this block get session.id = "my-js-session-id"
    });
    ```
    </CodeGroup>
  </Step>
</Steps>

<Note>
  For more on setting `session.id` with traceAI helpers and context, see the [manual tracing guide](/docs/observe/features/manual-tracing/set-session-user-id).
</Note>
