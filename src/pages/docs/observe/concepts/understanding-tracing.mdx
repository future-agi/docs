## About

Tracing is the mechanism that brings application execution data into Future AGI. Without tracing, there is nothing to observe, evaluate, or alert on.

When an application makes an LLM call, runs a tool, or executes a chain step, tracing captures that operation as a **span**. Spans record the input, output, execution time, token count, cost, and status. Related spans are grouped into a **trace**, which represents one complete request or execution. Traces are sent to Future AGI over HTTP or gRPC using the OpenTelemetry protocol (OTLP) and stored by project.

This data powers everything in the platform: the trace list, span detail view, [sessions](/docs/observe/features/session), [user analytics](/docs/observe/features/users), [evaluations](/docs/observe/features/evals), and [alerts](/docs/observe/features/alerts).

---

## Why tracing matters

- **Execution visibility**: See the full path of a request (spans in order, parent/child) to understand what the app did and where time or cost went.
- **Per-span metrics**: Latency, token usage, and cost are attached to spans so slow or expensive operations are easy to spot.
- **Debugging**: Tie a failure or bad output to a specific span and trace for reproduction and fixing.
- **Evals and alerts**: Evaluations and alerts run on observation spans. Tracing is what fills those spans.
- **Sessions and users**: When spans carry `session.id` or `user.id`, traces are grouped into sessions and end users in the UI.

## Two ways to instrument

### Auto-instrumentation
Install a framework-specific traceAI package (e.g. `traceAI-openai`, `traceAI-langchain`) and call `.instrument()`. Every LLM call made through that framework is captured as a span automatically, with no changes to application code. This is the fastest way to get started.

### Manual tracing
Use `FITracer` to create spans explicitly with decorators (`@tracer.chain`, `@tracer.agent`, `@tracer.tool`) or context managers (`tracer.start_as_current_span`). Manual tracing is for custom operations, non-standard workflows, or cases where auto-instrumentors do not cover the code path.

Both methods can be combined in the same application. Auto-instrumentation handles framework calls while manual tracing fills the gaps.

## How it fits together

```
Application code
    │
    ├─ Auto-instrumentor (captures spans automatically)
    │
    ├─ FITracer (manual spans with decorators/context managers)
    │
    └─ Context helpers (propagate session.id, user.id, metadata)
    │
    ▼
OpenTelemetry (OTLP over HTTP or gRPC)
    │
    ▼
Future AGI Platform
    ├─ Traces & Spans
    ├─ Sessions & Users
    ├─ Evaluations
    ├─ Alerts & Monitors
    └─ Dashboards
```

---

## Two project types

Tracing powers two modes in the platform:

- **[Observe](/docs/observe)**: Production monitoring with traces, evals, alerts, sessions, and user analytics.
- **[Prototype](/docs/prototype)**: Test and compare LLM configurations before production. Promote a winner.

Both use the same tracing pipeline. The project type (`ProjectType.OBSERVE` or `ProjectType.EXPERIMENT`) determines which features are available.
