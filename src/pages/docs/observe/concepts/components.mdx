## About

Observability in LLM applications relies on a structured framework that captures execution details at different levels. Individual operations are recorded as **spans**, grouped into **traces**, and organized into **projects**. The transport layer is **OpenTelemetry (OTLP)**. Understanding these components is essential before instrumenting an application.

---

## Spans

A span represents a single operation within an execution flow. It records the input, output, execution time, status, and metadata for one discrete step.

### Structure of a span

- **Identification and context**: Unique span ID, trace ID, and optional parent span ID that establish the hierarchy. May include a project reference.
- **Execution details**: Operation name, span type, input/output data. If the operation fails, error metadata captures failure details.
- **Timing and performance**: Start and end timestamps, latency, and resource usage (token count, cost).
- **Metadata and custom attributes**: Tags, annotations, and JSON-based extensible fields. Environment details like host, service instance, and deployment version.

### Span types

| Span Type | What it captures |
|-----------|-----------------|
| **LLM** | Model invocations, prompts, completions, token usage, invocation parameters |
| **Tool** | External tool or function calls with name, description, parameters, and output |
| **Chain** | Individual steps in a sequential workflow where data flows through connected operations |
| **Agent** | Actions by autonomous agents, including decision-making logic and tool interactions |
| **Retriever** | Data retrieval operations like database queries or document index lookups |
| **Embedding** | Text-to-vector transformations with model metadata and processing details |
| **Reranker** | Result reordering or ranking adjustments with input documents and updated scores |
| **Guardrail** | Compliance checks, validation results, and applied safety policies |
| **Evaluator** | Assessment activities measuring model performance with scoring data and feedback |
| **Unknown** | Fallback for operations that do not fit predefined types |

### Span attributes

Attributes are key-value pairs that carry metadata about the operation a span tracks (e.g. model name, token count, invocation parameters).

**Rules:**
- **Keys**: Must be non-null strings.
- **Values**: Must be one of: string, boolean, float, integer, or an array of these types.

traceAI defines [semantic conventions](/docs/observe/features/manual-tracing/semantic-conventions) that standardize attribute names across providers and frameworks.

## Traces

A trace connects multiple spans to represent the full execution flow of a single request. It provides a structured view of how different operations interact.

### What traces enable

- **Dependency analysis**: Understand how retrieval, inference, and tool execution relate to each other.
- **Performance monitoring**: Measure latency across spans to find bottlenecks.
- **Error diagnosis**: Follow the execution path from input to output to find where things went wrong.

### Example trace

A trace for an AI-powered search system might contain:

1. A **retriever span** fetching relevant documents.
2. An **LLM span** generating a response from those documents.
3. A **tool span** calling an external API to enrich the answer.

Each span records its own timing, input/output, and status. Together they show the complete request flow.

## Projects

A project is a container that groups traces for a specific application, use case, or deployment. Projects keep observability data organized and separated.

### What projects enable

- **Segmentation**: Separate traces for different applications (e.g. customer support AI vs. content generation AI).
- **Version comparison**: Track improvements across model versions within the same project.
- **Filtering and analysis**: Analyze execution trends across traces within a project.

### Two project types

| Type | Purpose |
|------|---------|
| **Observe** (`ProjectType.OBSERVE`) | Production monitoring with sessions, evals, alerts, and user analytics |
| **Experiment** (`ProjectType.EXPERIMENT`) | Prompt testing with eval tags and version names for comparing runs |

---

## OpenTelemetry

[OpenTelemetry (OTel)](https://opentelemetry.io/) is the open-source framework that handles trace collection and export. Future AGI uses OTel because it is vendor-neutral, open source, and handles traces at scale.

- Spans are exported over **HTTP** or **gRPC** using the OTLP protocol.
- traceAI builds on top of OTel, adding AI-specific semantic conventions and framework instrumentors.
- Any OTel-compatible backend can receive traces from traceAI, not just Future AGI.
