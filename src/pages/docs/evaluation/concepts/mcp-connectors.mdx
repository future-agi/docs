## About

An MCP connector is a registered link to an [MCP](https://modelcontextprotocol.io) server. When you attach connectors to an eval, the judge runs in **Agent mode**: it can call the connector's tools while it reasons about whether the output passes.

This lets an eval verify a model's response against ground truth that lives outside the dataset row. The judge looks up a record in your CRM, fetches a knowledge-base article, queries Sentry for a related error, or hits an internal API, then uses what it found to decide.

A concrete example:

| Eval input | Connector used | What the judge does |
|---|---|---|
| Customer reply quoting an order ID | Internal orders API | Calls `get_order(id)`, compares the reply against the real order state |
| Support agent answer about a feature | Notion docs MCP | Searches the docs for the cited feature, checks the answer matches |
| Code-review verdict from an LLM | GitHub MCP | Fetches the PR diff and verifies the verdict references real changes |

Without connectors the judge can only reason from what is in the prompt. With connectors it can verify against your real systems.

---

## When to use

- The output references an identifier (order, ticket, user, PR) and you want the judge to check the live record instead of trusting the model.
- The "correct" answer changes over time (live pricing, latest docs, current inventory) and you do not want to embed it in the dataset.
- You want one eval to score across multiple sources: docs, ticketing, monitoring, internal APIs.
- The dataset is large and copying ground truth into every row would be impractical or stale.

If the ground truth fits cleanly in the dataset row, use a regular eval (Groundedness, Context Adherence) instead. Connectors add latency and judge tokens.

---

## How connectors plug into an eval

1. You register an MCP server under [Falcon AI Connectors](/docs/falcon-ai/features/mcp-connectors). The same connector list powers chat, skills, and evals.
2. You create or edit an eval, switch its mode to **Agent**, and tick the connectors you want available.
3. At eval time the judge gets the tool schemas from each enabled connector. While scoring it can call any of those tools the same way it would in chat.
4. The eval result includes the score, the reason, and the trace of tool calls the judge made.

The eval still produces the normal output shape (pass/fail, choice, score). The connector calls are an internal step the judge takes before deciding.

---

## What the judge can see

| Layer | Visible to the judge |
|---|---|
| Eval inputs (dataset columns) | Yes — same as a regular eval |
| Rule prompt | Yes |
| Connector tool schemas | Yes — names, descriptions, parameters |
| Connector tool results | Yes — only for tools the judge actually invokes during the run |
| Other workspace data | No — connectors are scoped to the eval's workspace |

A connector that requires OAuth runs as the user who authenticated it. A connector that uses an API key or bearer token uses the same credentials for every eval row.

---

## Cost and latency

- **Tokens**: Each connector call adds the tool schema and the response to the judge's context. A 10-tool connector with verbose schemas can add several hundred tokens per row. Enable only the tools the eval needs.
- **Latency**: Each tool call is a network round trip. Evals that average 2-3 tool calls per row will run noticeably slower than a regular LLM-as-Judge eval. Plan for batch runs accordingly.
- **Rate limits**: A 10,000-row eval that hits a connector once per row makes 10,000 calls. Make sure the upstream service tolerates that, and use the connector's tool selection to keep traffic focused.

---

## Internet tool

Alongside MCP connectors, the Agent mode exposes a built-in **Internet** tool that lets the judge fetch a public web page. Enable it the same way you enable a connector. Useful for evals that need to verify a citation, check a public API spec, or compare an answer against a published source.

The Internet tool fetches one URL at a time and does not crawl. If you need indexed search across many pages, register an MCP search server instead.

---

## Limitations

- The judge decides which tools to call. You cannot force a specific tool sequence from the rule prompt. Write the prompt so the judge has a clear reason to use the tool.
- Connector failures do not fail the eval. The judge sees the error and decides whether to retry, try another tool, or score without the lookup. Watch the trace if scores look off.
- Sensitive data returned by a connector is included in the judge's reasoning trace. Do not expose connectors that read secrets.
