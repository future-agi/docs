## About

This page shows how to attach MCP connectors to an eval so the judge can call external tools while scoring. For the concept and trade-offs, see [MCP Connectors in Evaluation](/docs/evaluation/concepts/mcp-connectors). To register a new MCP server, see [Falcon AI MCP Connectors](/docs/falcon-ai/features/mcp-connectors).

---

## Prerequisites

- The MCP connector is already registered and tested in **Settings → Connectors**.
- You have a dataset, simulation run, or trace to evaluate.
- Your workspace has access to a judge model that supports tool calling. `turing_large` is the default.

---

## Create an Agent-mode eval with connectors

<Steps>
  <Step title="Open the eval picker">
    From the dataset, simulation, or trace view click **Evaluate**. The eval configuration panel opens on the right.

    ![Eval configuration panel](/screenshot/product/evaluation/mcp-connectors/1.png)

    <Note>Image placeholder — replace with the actual eval picker screenshot.</Note>
  </Step>

  <Step title="Add a custom eval or pick a template">
    Click **Add Evaluation**, then **Create your own eval** for a new template, or pick a built-in template that already runs in Agent mode (any **Customer Agent** template).

    ![Add a custom eval](/screenshot/product/evaluation/mcp-connectors/2.png)
    <Note>Image placeholder.</Note>
  </Step>

  <Step title="Switch the mode to Agent">
    In the eval config, set **Mode** to **Agent**. The **Tools** section appears below the rule prompt.

    | Mode | Tools available | When to use |
    |---|---|---|
    | Quick | None | Fast, single-shot LLM judgment from the prompt alone |
    | Agent | Connectors + Internet | The judge needs to look up live data while scoring |
  </Step>

  <Step title="Enable connectors">
    In **Tools**, toggle on the connectors you want this eval to use. Each connector lists the discovered tools below its name; you can disable individual tools to keep the judge focused.

    ![Connectors selection in eval](/screenshot/product/evaluation/mcp-connectors/3.png)
    <Note>Image placeholder.</Note>

    Toggle **Internet** on if the judge should be able to fetch public web pages.
  </Step>

  <Step title="Write the rule prompt">
    Make it explicit which tool the judge should consider using. The judge picks tools on its own, but a clear prompt gives it the reason to.

    ```text
    The model claims that order {{order_id}} has shipped.
    Call `orders.get_status(order_id)` and confirm the real status
    matches the claim. Pass if the statuses agree, fail otherwise.
    Quote the live status in your reason.
    ```

    Use `{{column_name}}` to reference dataset columns. The variables get substituted per row at eval time.
  </Step>

  <Step title="Pick the judge model">
    Choose a model that supports tool calling. `turing_large` is recommended; `turing_large_xl` handles long tool outputs better when the connector returns large payloads.

    ![Pick judge model](/screenshot/product/evaluation/mcp-connectors/4.png)
    <Note>Image placeholder.</Note>
  </Step>

  <Step title="Run on a single row first">
    Use **Test on Sample** with the eval picker open to score one row against your tools before you commit to the full run. The test panel shows the tool calls the judge made, the responses, and the final score.

    ![Test on sample](/screenshot/product/evaluation/mcp-connectors/5.png)
    <Note>Image placeholder.</Note>

    If the judge did not call the expected tool, refine the rule prompt. If the tool call failed, fix the connector configuration in Settings before running at scale.
  </Step>

  <Step title="Run at scale">
    Once the sample looks right, click **Add & Run** to score every row. Each row makes its own connector calls in parallel.

    <Warning>
      Each row makes its own tool calls. If your dataset has 10,000 rows and the judge averages two calls per row, the connector will receive ~20,000 requests. Check the upstream service's rate limits.
    </Warning>
  </Step>
</Steps>

## Reading the result

Each row's eval result contains:

| Field | What you see |
|---|---|
| **Score** | The eval output (Passed/Failed, a choice, or a numeric score). |
| **Reason** | The judge's explanation, citing the data it fetched. |
| **Tool Trace** | The list of tool calls and their responses. Click to expand. |

If the judge made no tool calls on a row where it should have, the reason will usually say so. Use that as a signal to sharpen the rule prompt.

## Common configurations

### Verify a model's claim against an internal record

```text
The model said: "{{output}}"
For the user with id {{user_id}}, call `crm.get_user(user_id)`
and confirm the response matches the user's current plan.
Pass if it does, fail with a one-line reason otherwise.
```

Pair with a `crm` MCP connector exposing read-only user lookups.

### Score answers against the latest docs

```text
The agent answered: "{{output}}"
Search Notion for the feature "{{feature_name}}".
Pass if the answer matches the latest doc content;
fail with the doc excerpt that contradicts it otherwise.
```

Pair with a Notion MCP server with `search` and `get_page` tools enabled.

### Cross-check a code review verdict against the diff

```text
The reviewer said the PR is safe to merge: "{{review_text}}"
Call `github.get_pr_diff({{pr_number}})` and assess whether the
diff contains any change that contradicts the reviewer's verdict.
Pass only if the diff supports the verdict.
```

Pair with a GitHub MCP connector with `get_pr_diff` enabled.

## Troubleshooting

| Symptom | What to check |
|---|---|
| Judge never calls the tool | Rule prompt does not mention what the tool is for. Spell out the lookup explicitly. |
| Tool returns auth error on every row | Re-authenticate the connector in **Settings → Connectors → Authenticate**. OAuth tokens expire. |
| Score is consistent but wrong | Tool selection may be too broad. Disable tools the eval shouldn't need so the judge picks the right one. |
| Eval runs very slowly | Each tool call is a network round trip. Lower the dataset size for the run, or split the eval into a cheaper pre-filter and an Agent-mode eval. |
| Connector returns large payloads that overflow context | Switch the judge to `turing_large_xl`, or narrow the tool's input parameters in the rule prompt. |
