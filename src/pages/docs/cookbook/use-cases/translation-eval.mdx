LinguaFlow uses an LLM to translate product strings into Spanish. Most translations come back fine. But then the error message uses "tu" instead of "usted," which is too informal for enterprise software. The technical docs translate "SSO" to "inicio de sesion unico" instead of keeping it as "SSO." A button label that is 12 characters in English becomes 30 characters in Spanish and overflows the UI. Each of these problems is different, and a generic translation quality score misses all of them.

The challenge is scale. Your product has hundreds of UI strings spread across button labels, error messages, tooltips, marketing copy, technical docs, and legal disclaimers. Each string type has different rules: formality register, character length limits, terms that must stay in English. Checking every string manually for even one target language takes weeks. Add more languages and the review backlog grows faster than your team can clear it.

What if you could automatically check semantic accuracy with a purpose-built translation metric, verify that the formality register matches each string type, ensure technical terms stay in English, enforce per-string-type character budgets, and run all four checks as a batch? You would see exactly which translations need human review and which are safe to auto-publish, with specific failure reasons attached to every flagged string.

This cookbook builds that evaluation pipeline for LinguaFlow's Spanish localization using FutureAGI's **Evals** ecosystem. You will translate a set of product strings with GPT-4o-mini, then run four layered evaluations: the **`translation_accuracy`** built-in metric for semantic preservation, plus three **custom Evals** for formality register, technical term retention, and UI character budgets. Each one catches a different failure mode that a generic translation quality score would miss.

<Step title="Translate a batch of product strings">

Start with what you already have: an LLM-based translator and a set of English strings that need localizing. This translator uses a system prompt specifying target language, formality level, and domain context.

```python
import os
from openai import OpenAI

client = OpenAI()

def translate(text: str, target_lang: str, string_type: str) -> str:
    """Translate a product string to the target language."""
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        temperature=0.3,
        messages=[
            {
                "role": "system",
                "content": f"""You are a professional software localization translator.
Translate the following product string from English to {target_lang}.

Rules:
- Use formal register (usted/Sie/vous) for UI strings, error messages, help text, and legal text
- Informal register (tú/du/tu) is acceptable for marketing copy only
- Technical terms (API, SDK, JSON, OAuth, SSO, URL, HTTP) must remain in English
- Keep translations concise. UI strings should not exceed 130% of the source length
- Preserve any placeholder variables like {{name}} or %s exactly as-is
- String type: {string_type}

Return ONLY the translated string, nothing else.""",
            },
            {"role": "user", "content": text},
        ],
    )
    return response.choices[0].message.content.strip()
```

<Step title="Check whether translations preserve meaning">

With references in place, start with the most important question: does the translation say the same thing as the source? `translation_accuracy` is a built-in metric designed specifically for this. It checks semantic accuracy, cultural appropriateness, and whether the original meaning is preserved in natural-sounding target language.

```python
from fi.evals import evaluate

for item in test_dataset:
    result = evaluate(
        "translation_accuracy",
        output=item["translation"],
        input=item["source"],
        model="turing_small",
    )

    status = "PASS" if result.passed else "FAIL"
    print(f"[{item['string_type']}]")
    print(f"  Source:      {item['source'][:60]}...")
    print(f"  Translation: {item['translation'][:60]}...")
    print(f"  Accuracy: score={result.score}  {status}")
    print(f"  Reason: {result.reason}\n")
```

<Step title="Verify formality register matches the string type">

Meaning preservation is necessary but not sufficient. A translation can be semantically accurate and still sound wrong if the formality level is off. Spanish has two registers: "usted" (formal) and "tu" (informal). Enterprise software should use "usted" for UI strings, but marketing copy can use "tu" to sound more approachable. A built-in metric can't know your formality rules, so you need a custom eval.

**Create the eval in the dashboard:**

1. Go to [app.futureagi.com](https://app.futureagi.com) -> **Evals** (left sidebar under BUILD)
2. Click **Create Evaluation**
3. Fill in:
   - **Name**: `translation_formality`
   - **Template type**: **Use Future AGI Agents**
   - **Model**: `turing_small`
   - **Output Type**: `Pass/Fail`

4. Write the **Rule Prompt**:

```
You are evaluating a Spanish translation for correct formality register.

Source text (English): {{source_text}}
Translation (Spanish): {{translated_text}}
String type: {{string_type}}

RULES:
- For string types "button_label", "error_message", "help_tooltip", "technical_docs", and "legal_disclaimer": the translation MUST use formal register ("usted", "su", conjugations like "verifique", "haga clic", "navegue").
- For string type "marketing_headline" or "marketing_copy": informal register ("tú", "tu", conjugations like "potencia", "descubre") is acceptable.
- Look for verb conjugations, possessive pronouns, and direct address to determine register.

Mark PASS if the translation uses the correct register for its string type.
Mark FAIL if formal text uses informal register, or if the register cannot be determined because the translation is too short or ambiguous (in which case, note that).
```

5. Click **Create Evaluation**

**Run the eval:**

```python
import os

    fi_api_key=os.environ["FI_API_KEY"],

for item in test_dataset:

        eval_templates="translation_formality",

```

<Step title="Enforce UI length constraints">

A translation can be accurate, formal, and preserve every technical term, and still break your product if it does not fit the layout. UI strings have layout budgets. A Spanish translation typically runs 20-30% longer than English. If a button label expands beyond what the layout allows, it overflows, wraps, or gets clipped. This eval enforces per-string-type length ceilings.

**Create the eval in the dashboard:**

1. Go to **Evals** -> **Create Evaluation**
2. Fill in:
   - **Name**: `translation_length_constraint`
   - **Template type**: **Use Future AGI Agents**
   - **Model**: `turing_small`
   - **Output Type**: `Pass/Fail`

3. Write the **Rule Prompt**:

```
You are evaluating whether a translated string is within acceptable length limits for UI display.

Source text (English): {{source_text}}
Translation: {{translated_text}}
String type: {{string_type}}

LENGTH RULES:
- For "button_label": translation must be within 130% of source character count.
- For "help_tooltip": translation must be within 130% of source character count.
- For "error_message": translation must be within 150% of source character count.
- For "marketing_headline": translation must be within 140% of source character count.
- For "technical_docs" and "legal_disclaimer": no strict length limit (PASS automatically).

Calculate the source length and translation length in characters. Compute the ratio (translation length / source length * 100).

Mark PASS if the ratio is within the allowed percentage for the string type.
Mark FAIL if the ratio exceeds the limit. Report both lengths and the ratio.
```

4. Click **Create Evaluation**

**Run the eval:**

```python
for item in test_dataset:

        eval_templates="translation_length_constraint",
        inputs={
            "source_text": item["source"],
            "translated_text": item["translation"],
            "string_type": item["string_type"],
        },

    eval_result = result.eval_results[0]
    source_len = len(item["source"])
    trans_len = len(item["translation"])
    ratio = trans_len / source_len * 100

    print(f"[{item['string_type']}]")
    print(f"  Source length:      {source_len} chars")
    print(f"  Translation length: {trans_len} chars ({ratio:.0f}%)")
    print(f"  Within limit: {eval_result.output}")
    print(f"  Reason: {eval_result.reason}\n")
```

<Step title="Run all checks as a batch and find weak spots">

Each eval checks one dimension of quality. Running all four together gives you a complete pass/fail verdict per string so you can route failures to human translators and auto-publish the rest. Upload the dataset, run built-in and custom evals together, and download the scored results.

import time

with open(csv_path, "w", newline="", encoding="utf-8") as f:

    writer.writeheader()

# Identify which string types need human review
eval_cols = [c for c in df.columns if "check" in c.lower() and "reason" not in c.lower()]

print("\n=== Translation Quality Summary ===\n")
for _, row in df.iterrows():
    string_type = row["string_type"]
    failures = []
    for col in eval_cols:
        val = row[col]
        if val in [False, "Failed", "failed", "Fail", "fail", 0, 0.0]:
            failures.append(col)

    status = "NEEDS REVIEW" if failures else "AUTO-PUBLISH OK"
    print(f"[{string_type}] {status}")
    if failures:
        for col in failures:
            reason_col = [c for c in df.columns if col.replace("-check", "") in c.lower() and "reason" in c.lower()]
            reason = row[reason_col[0]] if reason_col else "No reason available"
            print(f"  Failed: {col}")
            print(f"  Reason: {reason}")
    print()
