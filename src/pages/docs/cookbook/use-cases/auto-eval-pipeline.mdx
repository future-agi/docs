ReturnRight launched a RAG-powered support bot last week. Within days, a customer got quoted $1,899 for a laptop that actually costs $1,499. Another received a response containing someone else's email address. A wrong price turns into a refund. A leaked email turns into a support escalation and a trust problem. Both slipped through because there were no quality checks between the model and the customer.

ReturnRight handles hundreds of support conversations every day. Manually reviewing each one is not realistic. Picking the wrong eval metrics wastes compute and engineering time. Skipping the right ones lets exactly these kinds of bugs ship to production. You need a way to choose the right checks without becoming an expert in every available metric first.

What if you could describe your app in plain text and get back a recommended set of eval metrics and safety scanners tailored to your use case? Then customize the thresholds for your domain, run everything on real outputs to verify it catches actual bugs, and export the config so every deploy gets checked automatically in CI.

This cookbook sets up that pipeline for ReturnRight's support bot using FutureAGI's **AutoEvalPipeline**. You will use **Evals** templates and **Protect** scanners under the hood, tune them for your domain, and export the config as YAML so the same checks run as a quality gate in CI on every deploy. By the end, you go from zero quality gates to a tuned, exportable eval config in about 20 minutes.

<Steps>
<Step title="Describe ReturnRight's support bot and get recommended evals">

Tell the pipeline what your app does. Be specific about data sources, user interactions, and sensitive data. The more detail you give, the better the recommendations.

```python
from fi.evals.autoeval import AutoEvalPipeline

pipeline = AutoEvalPipeline.from_description(
    "A RAG-based customer support chatbot for an e-commerce platform. "
    "It retrieves product specs, return policies, and order history, "
    "then generates answers to customer questions. Customers sometimes "
    "share order IDs and email addresses in their messages."
)

print(pipeline.explain())
```

Sample output (your results may vary):
```
AutoEval Config: autoeval_customer_support
  Category: customer_support
  Risk Level: medium
  Domain: pii_sensitive

Evaluations (1):
  - answer_relevancy (threshold: 0.75, enabled)

Scanners (4):
  - JailbreakScanner (action: block, enabled)
  - ToxicityScanner (action: block, enabled)
  - PIIScanner (action: flag, enabled)
  - SecretsScanner (action: flag, enabled)
```

The pipeline detected a customer support app handling PII-sensitive data and recommended `answer_relevancy` for quality plus four safety scanners.

The pipeline analyzes your description and identifies:
- **App category** (e.g., RAG system, customer support, code assistant)
- **Risk level** based on the domain (general, healthcare, financial)
- **Domain sensitivity** (PII handling, compliance requirements)

From these, it selects evals that catch the failure modes your app is most likely to hit, and scanners that guard against safety risks.

<Step title="Tune the pipeline for ReturnRight's needs">

The `rag_system` template is a starting point with sensible defaults. Tune it by adjusting thresholds for your domain and adding scanners that match your risk profile. ReturnRight handles user-submitted text, so screening for secrets and invisible characters helps prevent injection and data leakage.

```python
from fi.evals.autoeval import EvalConfig, ScannerConfig

# Tighten thresholds: ReturnRight can't afford hallucinated prices
pipeline.set_threshold("faithfulness", 0.85)
pipeline.set_threshold("answer_relevancy", 0.8)
pipeline.set_threshold("groundedness", 0.8)

# Add secrets scanner to catch leaked API keys or tokens in responses
pipeline.add(ScannerConfig(
    name="SecretsScanner",
    action="flag",
    threshold=0.7,
))

# Add invisible character scanner to detect injection via hidden unicode
pipeline.add(ScannerConfig(
    name="InvisibleCharScanner",
    action="block",
    threshold=0.7,
))

# Check the updated config
print(pipeline.summary())
```

Sample output (your results may vary):
```
AutoEvalPipeline: rag_system
  Evaluations: 3
  Scanners: 3
  Risk Level: medium

  Eval: faithfulness              threshold: 0.85  (enabled)
  Eval: groundedness              threshold: 0.8  (enabled)
  Eval: answer_relevancy          threshold: 0.8  (enabled)

  Scanner: JailbreakScanner          action: block
  Scanner: SecretsScanner            action: flag
  Scanner: InvisibleCharScanner      action: block
```

All operations are chainable. You can disable evals and adjust thresholds in one expression:

```python
pipeline.disable("groundedness").set_threshold("faithfulness", 0.9)
```

</Step>
<Step title="Run the pipeline on ReturnRight's real outputs">

Now test against actual support bot outputs. These are three real conversations from ReturnRight's first week, including the pricing bug that triggered the customer complaint.

```python
test_cases = [
    {
        "query": "What's your return policy for electronics?",
        "response": (
            "You can return electronics within 30 days of purchase for a full refund, "
            "as long as the item is in its original packaging and unused."
        ),
        "context": (
            "Return policy: Electronics may be returned within 30 days of purchase "
            "for a full refund. Items must be unused and in original packaging. "
            "Opened software is non-refundable."
        ),
    },
    {
        "query": "Where is my order #12345?",
        "response": (
            "Your order #12345 shipped on March 10 and is expected to arrive by March 14. "
            "Here's your tracking number: 1Z999AA10123456784."
        ),
        "context": (
            "Order #12345: Shipped 2025-03-10 via UPS Ground. "
            "Tracking: 1Z999AA10123456784. Estimated delivery: 2025-03-14."
        ),
    },
    {
        "query": "Do you sell laptops with 64GB RAM?",
        "response": (
            "Yes! We carry several laptops with 64GB RAM. Our most popular is the "
            "ProBook X1 starting at $1,899, which also comes in a 128GB configuration."
        ),
        "context": (
            "ProBook X1: 64GB RAM, starts at $1,499. "
            "ProBook X1 Max: 128GB RAM, starts at $2,299. "
            "No other models offer 64GB or above."
        ),
    },
]

for i, tc in enumerate(test_cases):
    result = pipeline.evaluate(tc)
    status = "PASSED" if result.passed else "FAILED"
    print(f"\nCase {i+1}: {status} ({result.total_latency_ms:.0f}ms)")
    print(result.explain())
```

Sample output (your results may vary):
```
Case 1: FAILED (8000ms)
AutoEval Result: FAILED
Total Latency: 8000.27ms

Scanner Result: PASSED

Metric Results (2):
  faithfulness: PASSED (score: 1.00)
  answer_relevancy: FAILED (score: 0.24)

Case 2: FAILED (107ms)
AutoEval Result: FAILED
Total Latency: 106.93ms

Scanner Result: PASSED

Metric Results (2):
  faithfulness: PASSED (score: 1.00)
  answer_relevancy: FAILED (score: 0.37)

Case 3: FAILED (123ms)
AutoEval Result: FAILED
Total Latency: 123.41ms

Scanner Result: PASSED

Metric Results (2):
  faithfulness: FAILED (score: 0.40)
  answer_relevancy: FAILED (score: 0.35)
```

`faithfulness` works exactly as expected. Cases 1-2 score 1.00 because the responses are fully supported by the retrieved context. Case 3 scores 0.40 because the response claims $1,899 when the context says $1,499. That is a real hallucination, and faithfulness catches it.

`answer_relevancy` scores low across all three cases despite the responses clearly addressing each question. This is a limitation of the local NLI model (DeBERTa): it measures textual entailment between query and response, which is a different task from semantic relevancy. For more accurate relevancy scoring in production, use Turing models by specifying `model` in your EvalConfig.

The key takeaway: `faithfulness` is the high-value metric here. It correctly identifies the one case with a factual error while passing the two that are faithful to context.

</Step>
<Step title="Find the pattern behind ReturnRight's failures">

After running the pipeline, aggregate results to see where failures cluster.

```python
from collections import Counter

# Re-run all test cases to collect results for aggregation
all_results = [pipeline.evaluate(tc) for tc in test_cases]

pass_count = sum(1 for r in all_results if r.passed)
print(f"Overall: {pass_count}/{len(all_results)} passed\n")

# Check which metrics failed most often
metric_failures = Counter()
for r in all_results:
    for mr in r.metric_results:
        if not getattr(mr, "passed", True):
            metric_failures[getattr(mr, "eval_name", "unknown")] += 1

if metric_failures:
    print("Most common failures:")
    for metric, count in metric_failures.most_common():
        print(f"  {metric}: {count}/{len(all_results)} cases failed")
else:
    print("No metric failures detected.")
```

Sample output (your results may vary):
```
Overall: 0/3 passed

Most common failures:
  answer_relevancy: 3/3 cases failed
  faithfulness: 1/3 cases failed
```

The failure breakdown confirms the pattern: `answer_relevancy` fails uniformly (3/3) which signals a scoring calibration issue with the local model, not a real problem in your app. `faithfulness` fails only 1/3, pinpointing the exact case with the hallucinated price.

When you see a metric fail every case, ask whether it is a real issue or a model calibration problem. Uniform failure across obviously correct responses is usually the latter. For production pipelines, pair local metrics (fast, free) with Turing metrics (accurate, API-based) to get both speed and precision.

</Step>
<Step title="Save the pipeline config for CI">

Now that the pipeline catches ReturnRight's failure modes, export it so every PR gets checked automatically. The YAML file goes into your repo alongside the app code.

```python
# Export to YAML (human-readable, good for code review)
pipeline.export_yaml("eval_pipeline.yaml")

# Export to JSON (good for programmatic use)
pipeline.export_json("eval_pipeline.json")

# Reload later
reloaded = AutoEvalPipeline.from_yaml("eval_pipeline.yaml")
print(reloaded.summary())
```

Sample output (your results may vary):
```
AutoEvalPipeline: rag_system
  Evaluations: 3
  Scanners: 3
  Risk Level: medium
```

The YAML file captures every eval, scanner, threshold, and execution setting. Commit it to your repo so pipeline changes go through the same review process as code changes.
